# /dev 개발 파이프라인 플러그인 설계 v2

## 개요

사내 5개 서비스(달라, 클럽라이브, 여보야, 클럽5678, AI식단)의 iOS/Android 개발 워크플로우를 단일 명령어 `/dev`로 자동화하는 Claude Code 플러그인.

로드맵의 **자동화 영역**(코드 작성, 단위 테스트, UI 테스트)과 **최적화 영역**(기획 회의, 개발 회의, 통합 테스트)을 단계 기반 단일 플로우로 통합한다.

## 핵심 원칙

- **단일 진입점:** `/dev` 하나로 모든 워크플로우에 진입
- **초보자 친화:** 선택지와 안내 메시지로 자연스럽게 흐름을 따라감
- **플랫폼 독립 설계:** 공통 설계(spec)를 먼저 작성하고, 플랫폼별 구현은 각 프로젝트의 CLAUDE.md를 따름
- **단계별 선행조건 강제:** 산출물 기반 자동 검증으로 워크플로우 순서를 보장
- **산출물 중심:** 모든 단계의 결과는 Notion 또는 코드 커밋으로 추적 가능
- **스킬 생성은 `/skill-creator` 사용 필수**

## 1. `/dev init` — 초기 설정

최초 1회 실행. 이후 `/dev` 진입 시 저장된 설정을 읽어서 사용.

### 수집 항목 (순서대로 질문)

1. **서비스 선택** — 달라 / 클럽라이브 / 여보야 / 클럽5678 / AI식단
2. **플랫폼 선택** — iOS / Android
3. **작업자 이름** — 자유 입력
4. **Notion MCP 검증**
   - Notion MCP 서버가 설치되어 있는지 확인
   - Notion 계정이 연결(로그인)되어 있는지 확인
   - 실패 시: 설치/연결 방법 안내

### Notion 페이지 매핑

서비스별로 사용하는 Notion 페이지가 고정이므로 **서비스 선택 시 자동 결정**. 매핑 테이블은 플러그인 내부에 하드코딩:

```json
{
  "달라": {
    "design": "page-id-dalla-design",
    "qa": "page-id-dalla-qa"
  },
  "클럽라이브": { "design": "...", "qa": "..." },
  "여보야": { "design": "...", "qa": "..." },
  "클럽5678": { "design": "...", "qa": "..." },
  "AI식단": { "design": "...", "qa": "..." }
}
```

### 저장

`.claude/dev-config.json` (개인별 설정, git 제외):

```json
{
  "service": "달라",
  "platform": "iOS",
  "worker": "윤병은"
}
```

Notion 페이지 ID는 설정 파일에 저장하지 않음 (서비스명으로 매핑 테이블에서 조회).

### 재설정

`/dev init`을 다시 실행하면 기존 설정을 보여주고, 변경할 항목만 선택적으로 수정 가능.

## 2. `/dev` 전체 플로우

### 진입

```
/dev
 │
 ├─ 설정 파일 확인 → 없으면 init으로 안내
 │
 ├─ "달라 / iOS / 윤병은 님으로 진행합니다"
 │
 ├─ "작업번호를 입력해주세요" (예: DCL-1351)
 │   ├─ 기존 작업: 산출물 기반으로 완료 상태 자동 검증
 │   └─ 신규 작업: 새 작업 시작
 │
 ├─ 단계 메뉴 (자동 검증 결과 반영)
 │   ── 기획 단계 ──
 │   ✓ 1. 기획서 검토
 │   ✓ 2. UI 흐름도 작성
 │     3. 데이터 흐름도 작성
 │
 │   ── 개발 단계 ──
 │     4. 기능 신규 개발
 │     5. 기능 변경 및 고도화
 │     6. 버그 수정
 │
 │   ── 테스트 단계 ──
 │     7. 테스트 시나리오 작성 (QA용)
```

### 작업번호 형식

서비스별 접두사가 다르지만 공통 패턴: `영문대문자-숫자` (예: `DCL-1351`, `REQ-375`)

### 선행조건 체인

| 단계 | 선행조건 |
|---|---|
| 1. 기획서 검토 | 없음 |
| 2. UI 흐름도 | 1번 완료 |
| 3. 데이터 흐름도 | 1번 완료 |
| 4. 기능 신규 개발 | 1, 2, 3번 완료 |
| 5. 기능 변경 및 고도화 | 1, 2, 3번 완료 |
| 6. 버그 수정 | 1번 완료 (긴급 시 스킵 가능) |
| 7. 테스트 시나리오 | 4 또는 5 완료 |

### 완료 상태 추적 — 산출물 기반 자동 검증

| 단계 | 검증 방법 |
|---|---|
| 1. 기획서 검토 | Notion 작업번호 페이지 하위에 기획서 검토 서브페이지 존재 여부 |
| 2. UI 흐름도 | Notion 작업번호 페이지 하위에 UI 흐름도 서브페이지 존재 여부 |
| 3. 데이터 흐름도 | Notion 작업번호 페이지 하위에 데이터 흐름도 서브페이지 존재 여부 |
| 4. 기능 신규 개발 | git log에서 `[작업번호]` 패턴 커밋 존재 여부 (예: `git log --grep='\[DCL-1351\]'`) |
| 5. 기능 변경 및 고도화 | git log에서 `[작업번호]` 패턴 커밋 존재 여부 |
| 6. 버그 수정 | git log에서 `[작업번호]` 패턴 커밋 존재 여부 |
| 7. 테스트 시나리오 | Notion QA 보드 작업번호 하위에 QA 시트 서브페이지 존재 여부 |

### 미충족 시 동작

선행조건이 안 된 메뉴를 선택하면:

> "기능 신규 개발을 진행하려면 기획서 검토, UI 흐름도, 데이터 흐름도가 완료되어야 합니다. 어떤 단계부터 시작하시겠습니까?"

미완료 단계 목록을 보여주고 선택하게 함.

### 재작업 플로우

이미 완료된 단계를 다시 선택할 수 있음 (기획서 수정, 회의 결과 반영 등):

```
이미 완료된 단계 재선택 시
 → "기획서 검토가 이미 완료되어 있습니다. 기획서가 수정되었나요?"
 → "네" → 수정된 기획서로 재검토 진행
```

재작업 완료 시 후행 단계에 영향 안내:

```
 → "기획서가 변경되었습니다. 다음 산출물도 업데이트가 필요할 수 있습니다:"
 → "  - UI 흐름도 (✓ 완료 → ⚠ 확인 필요)"
 → "  - 데이터 흐름도 (✓ 완료 → ⚠ 확인 필요)"
 → "어떤 단계를 업데이트하시겠습니까?"
```

### 영향 전파 규칙

| 재작업 단계 | 확인 필요 표시되는 후행 단계 |
|---|---|
| 1. 기획서 검토 | 2, 3, 4, 5, 7 |
| 2. UI 흐름도 | 4, 5 |
| 3. 데이터 흐름도 | 4, 5 |
| 4/5. 기능 개발/변경 | 7 |

### 기존 산출물 처리 규칙

- Notion에 해당 서브페이지가 이미 존재하면 새로 생성하지 않고 **업데이트**
- 업데이트 시 기존 내용을 유지하며 변경 부분만 반영
- 논의사항/수정사항 섹션은 건드리지 않음 (회의 기록 보존)

### Notion 페이지 구조

```
[서비스 정책서/설계서 페이지]
  └── DCL-1351/                    # 작업번호 페이지
      ├── 기획서 검토                # 1번 산출물
      ├── UI 흐름도                 # 2번 산출물
      ├── 데이터 흐름도              # 3번 산출물
      ├── 공통 설계                  # 4/5번 산출물
      └── 버그 분석 리포트            # 6번 (범위 밖 버그 시)

[서비스 QA 보드 페이지]
  └── DCL-1351/
      └── QA 시트                   # 7번 산출물
```

### 로컬 작업 디렉토리

```
.dev-work/                          # gitignore 대상
  └── DCL-1351/
      └── spec.pdf                  # 원본 기획서 참조용
```

### Notion 공통 규칙

- 모든 서브페이지에 **담당자 필수** (init 설정의 작업자명 자동 입력)

## 3. 각 단계별 세부 설계

### 3-1. 기획서 검토

**입력:** PDF 기획서 경로

**검토 관점:**
- 화면 흐름 완전성 (진입 ~ 이탈까지 모든 경로)
- 각 화면의 상태 정의 (로딩, 빈 상태, 에러, 성공)
- 엣지 케이스 기술 여부 (네트워크 오류, 권한 거부, 데이터 없음 등)
- 인터랙션 명세 명확성 (탭, 스와이프, 길게 누르기 등의 동작과 결과)

**처리 흐름:**
1. PDF를 페이지 단위로 읽음
2. 검토 관점으로 분석
3. 결과를 사용자에게 보여주고 확인
4. Notion 작업번호 페이지 하위에 **기획서 검토** 서브페이지 생성
   - 담당자: 작업자
   - 내용: 페이지별 이슈 목록 (위치, 문제점, 권장 수정사항)
   - 이슈 없을 시: "검토 통과" 기록
5. 피드백 HTML → PDF 생성 (기획자에게 전달용)

**산출물:**
- Notion 서브페이지 (검증용 + 기록용)
- 피드백 PDF (기획자 전달용)

**사용 스킬:** `spec-review.md`, `feedback-generator.md`, `notion-writer.md`

### 3-2. UI 흐름도 작성

**선행조건:** 1번 기획서 검토 완료

**입력:** 검토를 거쳐 보완된 최종 기획서 (기획자가 피드백 반영 후 전달한 것)

**처리 흐름:**
1. 최종 기획서에서 화면 목록 추출
2. 화면 간 전환 관계 정리 (진입 조건, 이동 트리거, 뒤로가기 동작)
3. 엣지 케이스별 분기 포함 (에러 → 얼럿, 권한 거부 → 설정 이동 등)
4. Mermaid 다이어그램으로 작성
5. 사용자에게 보여주고 확인
6. Notion 작업번호 페이지 하위에 **UI 흐름도** 서브페이지 생성

**Notion 서브페이지 구조:**

```
[DCL-1351] UI 흐름도
├── 담당자: 윤병은
├── 상태: 작성완료 / 회의완료
│
├── 흐름도
│   └── Mermaid 다이어그램 + 화면별 상태 정의 요약
│
├── 논의사항
│   └── (개발회의에서 논의할 항목을 자유 작성)
│       예: "에러 팝업 닫은 후 어디로 이동하는지 기획서에 없음"
│
└── 수정사항
    └── (회의 결과 확정된 변경 내용 기록)
        예: "에러 팝업 닫기 → 이전 화면으로 복귀로 확정"
```

**산출물:** Notion 서브페이지

**사용 스킬:** `ui-flow.md`, `notion-writer.md`

### 3-3. 데이터 흐름도 작성

**선행조건:** 1번 기획서 검토 완료

**입력:** 검토를 거쳐 보완된 최종 기획서

**처리 흐름:**
1. 기획서에서 데이터 항목 추출 (사용자 입력, API 요청/응답, 로컬 저장 등)
2. 데이터의 생성 → 가공 → 표시 → 저장 흐름 정리
3. 각 파트별 책임 구분 (프론트엔드 / 백엔드 / 외부 서비스)
4. Mermaid 다이어그램으로 작성
5. 사용자에게 보여주고 확인
6. Notion 작업번호 페이지 하위에 **데이터 흐름도** 서브페이지 생성

**Notion 서브페이지 구조:**

```
[DCL-1351] 데이터 흐름도
├── 담당자: 윤병은
├── 상태: 작성완료 / 회의완료
│
├── 흐름도
│   └── Mermaid 다이어그램 + 데이터 모델 정의 + API 엔드포인트 초안
│
├── 논의사항
│   └── (개발회의에서 논의할 항목)
│       예: "유저 프로필 이미지 URL, 서버에서 full URL로 주는지 path만 주는지?"
│
└── 수정사항
    └── (회의 결과 확정된 변경 내용)
        예: "프로필 이미지는 CDN full URL로 제공, 클라이언트 가공 없음"
```

**산출물:** Notion 서브페이지

**사용 스킬:** `data-flow.md`, `notion-writer.md`

### 3-4. 기능 신규 개발

**선행조건:** 1, 2, 3번 완료 + 개발회의 수정사항 반영 완료

**입력:** Notion의 UI 흐름도, 데이터 흐름도, 최종 기획서

**처리 흐름:**
1. **공통 설계 작성** — Notion 산출물들을 읽어서 플랫폼 무관한 설계 작성
   - 화면 구조, 데이터 모델, 상태 관리, API 연동 정의
   - 비즈니스 로직 명세 + 엣지 케이스 처리
   - 사용자 확인 후 Notion 작업번호 하위에 **공통 설계** 서브페이지 생성
2. **설계 승인** — 사용자가 공통 설계를 확인하고 승인
3. **TDD 테스트 작성** — 공통 설계 기반으로 단위/UI 테스트 먼저 작성
   - 프로젝트 CLAUDE.md의 아키텍처/컨벤션에 따름
   - superpowers `test-driven-development` 스킬 활용
4. **플랫폼별 구현** — 테스트를 통과시키는 방향으로 코드 작성
   - 구현 중 CLAUDE.md 기반 컨벤션/아키텍처 실시간 가이드
5. **코드리뷰** — Claude 자동 리뷰 → 사용자 최종 확인
   - superpowers `requesting-code-review` 스킬 활용
6. **커밋** — 작업번호를 커밋 메시지에 포함 (예: `[DCL-1351] 프로필 개편 기능 구현`)

**사용 스킬:** `common-design.md`, `implement.md`, `code-review.md`, `notion-writer.md`

### 3-5. 기능 변경 및 고도화

**선행조건:** 1, 2, 3번 완료

**입력:** Notion 산출물들 + 최종 기획서

**처리 흐름:**
1. **기존 설계 파악**
   - Notion에 기존 정책서/설계서가 있는 경우: Notion MCP로 조회
   - 없는 경우: 기존 코드를 분석하여 현재 설계 파악
2. **공통 설계 작성** — 기존 설계 대비 **변경점 중심**으로 작성
   - Notion 작업번호 하위에 **공통 설계** 서브페이지 생성
3. **설계 승인**
4. **TDD 테스트 작성** — 변경 부분에 대한 테스트 작성 + 기존 테스트 회귀 확인
5. **플랫폼별 구현** — CLAUDE.md 컨벤션 기반 실시간 가이드
6. **코드리뷰**
7. **커밋**

4번 신규 개발과의 차이점: **1단계에서 기존 설계를 먼저 파악**하고, 설계와 테스트 모두 변경점 중심으로 접근.

**사용 스킬:** `common-design.md`, `implement.md`, `code-review.md`, `notion-writer.md`

### 3-6. 버그 수정

**선행조건:** 1번 완료 (긴급 시 스킵 가능 — 사용자에게 확인)

**입력:** 사용자가 제공하는 버그 정보

**처리 흐름:**
1. **대화형 정보 수집** — 하나씩 질문하며 확보 (작업번호는 `/dev` 진입 시 이미 입력됨)
   - 에러코드 / 에러 메시지
   - 스크린샷
   - 상황 설명 (어떤 화면에서, 어떤 동작 시)
   - 재현 방법
   - 발생 빈도 (항상 / 간헐적)
2. **버그 유형 분류**
   - 로직/크래시 버그: 앱 크래시, 잘못된 동작, 데이터 불일치
   - UI 버그: 레이아웃 깨짐, 잘못된 표시, 애니메이션 이상
3. **원인 파트 특정**
   - API/서버 데이터 문제
   - 비즈니스 로직 오류
   - UI 상태 관리 오류
   - 데이터 바인딩/레이아웃 문제
   - 플랫폼 특이 이슈
4. **범위 판단 후 분기**
   - **개발 범위 내:**
     - superpowers `systematic-debugging` 스킬로 원인 분석
     - TDD 방식: 버그를 재현하는 테스트 먼저 작성 → 수정 → 테스트 통과 확인
     - 코드리뷰
     - 커밋 (예: `[REQ-375] 프로필 이미지 로드 실패 수정`)
   - **개발 범위 밖 (서버 등):**
     - 원인 리포트 생성: 문제 파트, 근거, 권장 조치
     - Notion 작업번호 하위에 **버그 분석 리포트** 서브페이지 생성

**사용 스킬:** `bug-fix.md`, `code-review.md`, `notion-writer.md`

### 3-7. 테스트 시나리오 작성 (QA용)

**선행조건:** 4 또는 5 완료

**입력:** 기획서 + Notion 공통 설계 + 구현된 코드

**처리 흐름:**
1. 기획서의 기능 요구사항에서 테스트 케이스 도출
2. 공통 설계의 엣지 케이스 / 상태 정의에서 추가 케이스 도출
3. 구현 코드에서 분기 조건 분석하여 누락 케이스 보완
4. 테스트 시나리오를 사용자에게 보여주고 확인
5. Notion QA 보드 작업번호 하위에 **QA 시트** 서브페이지 생성
   - 담당자: 작업자
   - 각 케이스: 시나리오명, 사전조건, 테스트 절차, 기대 결과, 우선순위

**Notion QA 시트 구조:**

```
[DCL-1351] QA 시트
├── 담당자: 윤병은
├── 플랫폼: iOS
│
├── 정상 케이스
│   ├── TC-01: 프로필 사진 변경 성공
│   │   사전조건 / 절차 / 기대 결과 / 우선순위: 높음
│   └── ...
│
├── 엣지 케이스
│   ├── TC-05: 네트워크 끊긴 상태에서 저장 시도
│   └── ...
│
└── 회귀 케이스
    ├── TC-10: 기존 프로필 조회 정상 동작 확인
    └── ...
```

**사용 스킬:** `qa-scenario.md`, `notion-writer.md`

## 6. v1 대비 변경 사항 요약

| 항목 | v1 | v2 |
|---|---|---|
| 메뉴 구조 | 4개 (기능 기준) | 7개 (단계 기준: 기획/개발/테스트) |
| 초기 설정 | 없음 | `/dev init` (서비스, 플랫폼, 작업자, Notion 검증) |
| 상태 추적 | 자기 신고 | 산출물 기반 자동 검증 |
| 선행조건 | 기획서 검토만 | 전 단계 체인 + 영향 전파 |
| UI/데이터 흐름도 | 없음 | 기획 단계에 추가 (Mermaid) |
| Notion 연동 | 읽기만 (기존 설계 조회) | 읽기 + 쓰기 (모든 산출물, 담당자 필수) |
| 테스트 | "테스트 작성 및 실행" 한 줄 | TDD 방식 (개발) + QA 시나리오 Notion 생성 (테스트) |
| 코드리뷰 | 완료 후 1회 | 구현 중 실시간 가이드 + 최종 리뷰 |
| 재작업 | 없음 | 완료된 단계 재진입 + 후행 단계 영향 전파 |
| 멀티서비스 | 없음 | 5개 서비스 + 작업번호 기반 관리 |
| 로드맵 커버리지 | 자동화 일부 | 자동화 + 최적화 전체 |
