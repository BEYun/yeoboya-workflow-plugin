# /dev 개발 파이프라인 플러그인 설계

## 개요

사내 3개 네이티브 앱(iOS/Android)의 개발 워크플로우를 단일 명령어 `/dev`로 자동화하는 Claude Code 플러그인.
팀 전체(Claude Code 초보자 포함)가 사용하는 표준 워크플로우로 도입한다.

## 핵심 원칙

- **단일 진입점:** `/dev` 하나로 모든 워크플로우에 진입
- **초보자 친화:** 선택지와 안내 메시지로 자연스럽게 흐름을 따라감
- **플랫폼 독립 설계:** 공통 설계(spec)를 먼저 작성하고, 플랫폼별 구현은 각 프로젝트의 CLAUDE.md를 따름
- **앱별 CLAUDE.md:** 3개 앱의 아키텍처/컨벤션이 다르므로 각 프로젝트의 CLAUDE.md에 정의

## 전체 구조

```
/dev
 │
 ├─ "무엇을 하시겠습니까?"
 │   1. 기획서 검토
 │   2. 기능 신규 개발
 │   3. 기능 변경 및 고도화
 │   4. 버그 수정
 │
 ├─ [1] 기획서 검토
 │   PDF 입력 → 페이지별 UI/UX 명세 리뷰 → 피드백 HTML→PDF 생성
 │
 ├─ [2/3/4 선택 시]
 │   → "기획서 검토 완료 여부" 확인
 │   → 미완료: 1번부터 안내 / 완료: 진행
 │
 ├─ [2] 기능 신규 개발
 │   기획서 → 공통 설계 → 플랫폼별 구현 → 테스트 → 코드리뷰
 │
 ├─ [3] 기능 변경 및 고도화
 │   기획서 → 기존 설계 파악(노션 MCP or 코드 분석)
 │   → 공통 설계 → 구현 → 테스트 → 코드리뷰
 │
 └─ [4] 버그 수정
     대화형 정보 수집
     → 버그 유형 분류 (로직/크래시 vs UI)
     → 원인 파트 특정 (API, 비즈니스 로직, UI 상태 등)
     → 개발 범위 내: 수정 → 테스트 → 코드리뷰
     → 개발 범위 밖(서버 등): 원인 리포트 제공
```

## 단계별 세부 설계

### 1. 기획서 검토

**입력:** PDF 파일 경로

**검토 관점:** UI/UX 명세 완성도
- 화면 흐름이 완전한가 (진입 ~ 이탈까지 모든 경로)
- 각 화면의 상태 정의가 충분한가 (로딩, 빈 상태, 에러, 성공)
- 엣지 케이스가 기술되어 있는가 (네트워크 오류, 권한 거부, 데이터 없음 등)
- 인터랙션 명세가 명확한가 (탭, 스와이프, 길게 누르기 등의 동작과 결과)

**처리 흐름:**
1. PDF를 페이지 단위로 읽음
2. 각 페이지의 UI/UX 명세 완성도를 체크
3. 이슈 발견 시: 페이지별 피드백이 담긴 HTML 생성 → PDF로 변환
4. 이슈 없음: "검토 통과" 안내, 다음 단계(2/3/4) 선택 유도

**피드백 PDF 구조:**
- 페이지 번호 기준으로 정리
- 각 이슈에 대해: 위치, 문제점, 권장 수정사항 포함
- 기획자에게 직접 전달 가능한 형태

### 2. 기능 신규 개발

**전제조건:** 기획서 검토 완료

**흐름:**
1. **기획서 참조:** 검토 완료된 PDF 기획서를 기반으로 요구사항 파악
2. **공통 설계(spec) 작성:** 플랫폼 무관한 설계 — 화면 구조, 데이터 모델, 화면 흐름, 상태 관리, API 연동 정의
3. **설계 승인:** 사용자 확인 후 진행
4. **플랫폼별 구현:** 현재 프로젝트의 CLAUDE.md에 정의된 아키텍처/컨벤션에 따라 구현
5. **테스트 작성 및 실행**
6. **코드리뷰:** Claude 자동 리뷰 → 사람 최종 확인

### 3. 기능 변경 및 고도화

**전제조건:** 기획서 검토 완료

**흐름:**
1. **기획서 참조:** 변경 요구사항 파악
2. **기존 설계 파악 (선제 작업):**
   - 노션 정책서/설계서가 있는 경우: 노션 MCP를 통해 읽고 분석
   - 없는 경우: 기존 코드를 분석하여 현재 설계 파악
3. **공통 설계(spec) 작성:** 기존 설계 대비 변경점 중심으로 작성
4. **설계 승인**
5. **플랫폼별 구현:** CLAUDE.md 컨벤션에 따름
6. **테스트 작성 및 실행**
7. **코드리뷰:** Claude 자동 리뷰 → 사람 최종 확인

### 4. 버그 수정

**전제조건:** 기획서 검토 완료 (단, 긴급 버그의 경우 기획서 없이 진행 가능 — 사용자에게 확인)

**흐름:**
1. **대화형 정보 수집:** 사용자에게 질문하며 필요한 정보 확보
   - 에러코드 / 에러 메시지
   - 스크린샷
   - 상황 설명 (어떤 화면에서, 어떤 동작 시)
   - 재현 방법
   - 발생 빈도 (항상 / 간헐적)
2. **버그 유형 분류:**
   - **로직/크래시 버그:** 앱 크래시, 잘못된 동작, 데이터 불일치
   - **UI 버그:** 레이아웃 깨짐, 잘못된 표시, 애니메이션 이상
3. **원인 파트 특정:**
   - API/서버 데이터 문제
   - 비즈니스 로직 오류
   - UI 상태 관리 오류
   - 데이터 바인딩/레이아웃 문제
   - 플랫폼 특이 이슈
4. **범위 판단:**
   - 개발 범위 내: 수정 → 테스트 → 코드리뷰
   - 개발 범위 밖(서버 등): 원인 리포트 제공 (문제 파트, 근거, 권장 조치)

## 플러그인 구조

```
yeoboya-workflow-plugin/
├── .claude-plugin/
│   └── plugin.json                # 플러그인 등록 (스킬 목록, 의존성)
├── skills/
│   └── dev/
│       └── dev.md                 # /dev 진입점 스킬
├── shared/
│   ├── spec-review.md             # 기획서 검토 로직
│   ├── feedback-generator.md      # 피드백 HTML→PDF 생성
│   ├── common-design.md           # 공통 설계 작성 가이드
│   ├── implement.md               # 플랫폼별 구현 가이드
│   ├── bug-classifier.md          # 버그 유형 분류 및 파트 특정
│   └── code-review.md             # 코드리뷰 가이드
└── docs/
    └── superpowers/
        └── specs/                 # 설계 문서
```

---

## 외부 의존성

- **superpowers 플러그인:** brainstorming, writing-plans, test-driven-development, requesting-code-review, receiving-code-review, systematic-debugging, verification-before-completion 스킬 활용
- **노션 MCP:** 기능 변경 시 기존 정책서/설계서 조회
- **HTML→PDF 변환:** 피드백 문서 생성 시 사용 (puppeteer, wkhtmltopdf 등 CLI 도구)

## 사용 시나리오

### 시나리오 1: 기획서 검토 → 반려
```
사용자: /dev
Claude: 무엇을 하시겠습니까?
        1. 기획서 검토
        2. 기능 신규 개발
        3. 기능 변경 및 고도화
        4. 버그 수정
사용자: 1
Claude: PDF 기획서 경로를 알려주세요.
사용자: ~/Documents/feature-spec.pdf
Claude: (페이지별 분석 후) 3건의 이슈가 발견되었습니다.
        - p.3: 에러 상태 UI 미정의
        - p.5: 화면 전환 시 뒤로가기 동작 미기술
        - p.7: 빈 목록 상태 미정의
        피드백 PDF를 생성했습니다: ~/Documents/feature-spec-feedback.pdf
```

### 시나리오 2: 기능 신규 개발
```
사용자: /dev
Claude: 무엇을 하시겠습니까? ...
사용자: 2
Claude: 기획서 검토가 완료되었나요?
사용자: 네
Claude: 검토 완료된 기획서 PDF 경로를 알려주세요.
사용자: ~/Documents/feature-spec.pdf
Claude: (기획서 분석 → 공통 설계 작성 → 승인 → 구현 → 테스트 → 코드리뷰)
```

### 시나리오 3: 버그 수정
```
사용자: /dev
Claude: 무엇을 하시겠습니까? ...
사용자: 4
Claude: 기획서 검토가 완료되었나요?
사용자: 네
Claude: 버그에 대해 알려주세요. 다음 정보가 있으면 공유해주세요:
        - 에러코드나 에러 메시지
        - 스크린샷
        - 어떤 화면에서 어떤 동작 시 발생하는지
        - 재현 방법
사용자: (정보 제공)
Claude: 분석 결과:
        - 유형: 로직 버그
        - 원인 파트: API 응답 데이터 파싱
        - (수정 → 테스트 → 코드리뷰 진행)
```
