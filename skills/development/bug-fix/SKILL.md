# bug-fix

버그 정보를 대화형으로 수집하고, 유형 분류 → 원인 특정 → 범위 판단 후 수정하거나 리포트를 생성한다.

---

## 선행조건

1단계(기획서 검토)가 완료되어야 한다. 단, 긴급 버그는 스킵 가능:

```
기획서 검토(1단계)가 완료되어 있나요?
긴급 버그라면 건너뛰고 바로 진행할 수 있습니다.
```

작업번호는 `/dev` 진입 시 사용자가 입력한 값을 사용한다. 이 스킬이 단독으로 호출된 경우 사용자에게 직접 질문한다.

---

## 0. 버그 수정 컨텍스트 선택

정보 수집 전에 **이 버그 수정의 성격**을 먼저 확인한다. 같은 "버그 수정"이라도 브랜치 전략과 커밋 규칙이 달라지기 때문이다.

```
[Q0] 이 버그 수정의 성격을 선택해주세요.
  (A) 현재 작업 중인 브랜치 내 수정
      — 진행 중인 작업에서 발견된 버그. 같은 브랜치/같은 작업번호로 커밋.
  (B) 독립 버그 수정
      — 기존 작업과 무관한 버그. 새 브랜치를 생성하고 별도 작업번호로 진행.
```

### 분기별 처리

**(A) 현재 브랜치 내 수정**
- 현재 브랜치를 그대로 사용한다. 별도 체크아웃 금지.
- `/dev` 진입 시 입력된 작업번호를 그대로 사용한다.
- 커밋 메시지는 현재 작업번호 기준. (§4-A 커밋 규칙 그대로)

**(B) 독립 버그 수정**
- 별도 작업번호가 필요하다. 없으면 사용자에게 요청한다.
- 새 브랜치를 생성한다:
  ```bash
  git checkout -b fix/<작업번호>-<간단한-슬러그>
  # 예: git checkout -b fix/DCL-1420-cake-deduction
  ```
- 브랜치 생성 전 현재 브랜치의 미커밋 변경이 있으면 사용자에게 먼저 확인한다 (stash/commit 여부).

선택 결과는 이후 단계 전반에 영향을 주므로 명확히 기록하고 진행한다.

---

## 1. 대화형 정보 수집

**한 번에 하나씩** 질문한다. "없음"/"모름" 허용.

```
[Q1] 에러코드나 에러 메시지가 있나요?
[Q2] 스크린샷이 있나요? (경로를 알려주세요)
[Q3] 어떤 화면에서, 어떤 동작 시 발생하나요?
[Q4] 재현 방법을 알려주세요.
[Q5] 항상 발생하나요, 간헐적으로 발생하나요?
```

충분한 정보가 모이면 나머지를 건너뛰고 분석으로 진행할 수 있다.

---

## 1.5 근본 원인 분석 (선행 게이트)

대화형 수집을 마친 직후, 유형 분류·원인 파트 특정에 진입하기 전에
`superpowers:systematic-debugging` 스킬을 호출하여 체계적으로 원인을 분석한다.

systematic-debugging의 산출(가설·증거·재현)이 이후 단계의 입력이 된다:
- 유형 분류(§2): 가설 카테고리 기반
- 원인 파트 특정(§3): 증거로 좁힌 영역
- 범위 내 수정(§4-A)의 TDD 수정: 재현 테스트로 직접 사용

---

## 2. 버그 유형 분류

| 유형 | 설명 | 예시 |
|------|------|------|
| Logic/Crash | 앱 크래시, 잘못된 동작, 데이터 불일치 | NullPointerException, 잘못된 계산 |
| UI | 레이아웃 깨짐, 잘못된 표시, 애니메이션 이상 | 버튼 위치 이상, 텍스트 잘림 |

---

## 3. 원인 파트 특정

| 원인 영역 | 판단 기준 | 처리 방향 |
|-----------|-----------|-----------|
| API/서버 데이터 | 서버 응답 이상, 잘못된 데이터 | 범위 밖 → 리포트 |
| 비즈니스 로직 | UseCase/Repository 버그 | 범위 내 → 수정 |
| UI 상태 관리 | ViewModel 상태 오류 | 범위 내 → 수정 |
| 데이터 바인딩/레이아웃 | View 바인딩 오류, 깨짐 | 범위 내 → 수정 |
| 플랫폼 특정 이슈 | 특정 OS/SDK에서만 발생 | 범위 내 → 수정 |

## 4-A. 범위 내 수정

### 근본 원인 분석
§1.5에서 수행한 `superpowers:systematic-debugging` 산출을 사용한다 (재실행 불필요).

### TDD 수정
1. **Red** — 버그를 재현하는 실패 테스트 작성
2. **Green** — 테스트를 통과시키는 최소 수정
3. **Verify** — 전체 테스트 실행, 기존 테스트도 통과 확인

**리뷰 전 커밋 금지** — TDD 사이클이 내부 커밋을 유도하더라도 이 스킬 범위에서는 보류하고 `git add`까지만 수행한다.

### 완료 전 검증
`superpowers:verification-before-completion` 스킬을 Skill 도구로 호출하여 "수정 완료" 주장 전 검증을 수행한다.

### 코드리뷰
`skills/development/code-review` 스킬을 Skill 도구로 호출하여 수정된 코드를 리뷰한다. 리뷰 통과 후에만 다음 단계로 진행한다.

### 커밋 (리뷰 통과 후)
리뷰에서 수정이 발생했어도 단일 커밋으로 정리한다:

```
[작업번호] [버그 설명] 수정
예: [DCL-1351] 케이크 생성 시 재료 차감 안 되는 버그 수정
```

---

## 4-B. 범위 밖 리포트

서버/외부 문제일 때 원인 리포트를 생성한다:

```markdown
## 버그 분석 리포트

### 문제 영역
[원인 영역 및 구체적 위치]

### 증거
[에러 메시지, 스크린샷, 재현 조건]

### 재현 단계
1. [단계]
2. [단계]

### 권장 조치
[서버팀/외부 담당자에게 요청할 구체적 액션]
```

`skills/common/notion-writer` 스킬을 사용하여 작업번호 페이지 하위에 **"버그 분석 리포트"** 서브페이지로 저장한다.
