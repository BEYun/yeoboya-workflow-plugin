# tech-spec

UI 흐름도와 데이터 흐름도를 종합하여 플랫폼 무관한 공통 설계를 작성한다. iOS/Android 개발자가 이 설계를 기반으로 각 플랫폼에 맞게 구현한다.

---

## 선행조건

2.1, 3.1, 3.2 스텝 모두 완료되어야 한다:
- 기획서 검토 (2.1)
- UI 흐름도 (3.1)
- 데이터 흐름도 (3.2)

**skills/common/notion-writer** 스킬로 세 서브페이지 존재 여부를 확인한다. 또한 개발회의 수정사항이 반영되었는지 사용자에게 확인한다:

```
UI 흐름도와 데이터 흐름도의 개발회의 수정사항이 반영되었나요?
```

---

## 두 가지 모드

### 모드 A: 신규 개발 (4단계)

새로운 기능을 처음 만들 때.

### 모드 B: 기능 변경/고도화 (5단계)

기존 기능을 수정하거나 개선할 때. 기존 설계를 먼저 파악하고 변경점 중심으로 작성한다.

모드는 `.dev-work/<작업번호>/state.json`의 `workType`에서 결정된다:

- `workType === "new"` → 모드 A (신규 개발)
- `workType === "change"` → 모드 B (기능 변경/고도화)
- `workType === "bugfix"` → 해당 파이프라인은 /dev가 3.x 전체를 스킵하므로 tech-spec은 호출되지 않는다

`workType`이 비어 있으면 /dev 라우터가 먼저 `work-define` 스킬을 호출하도록 안내하고 종료:
```
state.workType이 비어 있습니다. /dev에서 1.1 단계(작업 정의)를 먼저 완료해주세요.
```

## 모드 A: 신규 개발 설계

### 입력 수집

Notion에서 산출물을 읽는다 (**skills/common/notion-writer** 스킬 사용):
1. UI 흐름도 서브페이지 내용 (수정사항 섹션 포함 — 개발회의 결정 반영)
2. 데이터 흐름도 서브페이지 내용 (수정사항 섹션 포함)
3. 최종 기획서 PDF

### 설계 문서 구조

```markdown
## [작업번호] 공통 설계

### 1. 화면 구조
UI 흐름도 기반. 각 화면의 구성요소, 레이아웃 구조, 네비게이션.

### 2. 데이터 모델
데이터 흐름도 기반. 엔티티 정의, 관계, 타입.

### 3. 상태 관리
각 화면의 상태 정의와 전환 로직.
- 로딩 → 성공/실패 분기
- 사용자 액션별 상태 변경

### 4. API 연동
데이터 흐름도의 API 엔드포인트 + 요청/응답 상세.
경로는 /도메인명/... 형식.

### 5. 비즈니스 로직
핵심 비즈니스 규칙과 검증 로직.
- 입력 검증 규칙
- 계산/변환 로직 (예: 보너스 달 배율 계산)
- 권한/제한 체크 (예: 1일 1케이크 선물 제한)

### 6. 엣지 케이스 처리
기획서 + UI/데이터 흐름도에서 도출된 예외 상황과 대응.
```

## 모드 B: 기능 변경 설계

### 기존 설계 파악 (추가 단계)

먼저 기존 설계를 파악한다:
1. Notion에 기존 정책서/설계서가 있는지 **skills/common/notion-writer**로 검색
2. 있으면 → 기존 설계 내용 조회
3. 없으면 → 기존 코드를 분석하여 현재 설계 파악 (프로젝트 CLAUDE.md 참조)

### 설계 문서 구조 (변경점 중심)

```markdown
## [작업번호] 공통 설계 (변경)

### 1. 변경 배경
왜 변경이 필요한지.

### 2. 기존 설계 요약
현재 동작 방식 간단 정리.

### 3. 변경 사항
각 섹션별 AS-IS → TO-BE.

| 항목 | AS-IS | TO-BE |
|------|-------|-------|
| 데이터 모델 | ... | ... |
| API | ... | ... |
| 비즈니스 로직 | ... | ... |

### 4. 영향 범위
변경으로 인해 함께 수정해야 하는 부분.
```
