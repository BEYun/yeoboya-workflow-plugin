# 여보야 사내 플러그인 개발 로드맵
#클로드코드

# 개요 및 목표
- 여러 서비스(달라, 클럽라이브, 여보야, 클럽5678, 식단AI)를 개발하는 솔루션개발부(iOS/Android)에서 공통적으로 작업중인 업무 파이프라인을 클로드 코드 플러그인을 접목하여 **자동화 및 최적화**하는 것
- **자동화 영역**(솔루션개발부 작업), **최적화 영역**(타 부서 단독 또는 협업 작업)을 분리하여 현실적인 로드맵 제시

# 업무 파이프라인 자동화와 최적화 가능 범위
### 1. 자동화 가능 범위
- 코드 작성
- 단위 테스트
- UI 테스트

### 2. 최적화 가능 범위
- 기획 회의
- 개발 회의
- 통합 테스트

## 문제점 정의

### 근본 원인
==🟡**"코드 작성과 검증을 전적으로 개발자 개인의 역량과 판단에 의존한다"**==
구현 기준, 코드 품질, 테스트 커버리지를 보장하는 자동화 장치가 없기 때문에,
개발자마다 코드 품질이 달라지고 버그가 배포 후에야 발견되는 구조가 반복됨.

### 문제 1(구현) — 플랫폼 간 구현 불일치
iOS · Android · Web이 동일한 기능을 각자의 방식으로 구현.
공통 로직·예외 처리 기준이 없어 플랫폼마다 동작이 달라짐.
==🔴**결과:** 특정 플랫폼에서만 발생하는 버그, 플랫폼 간 재작업 루프==

### 문제 2(품질) — 코드 품질 기준 부재
코드 리뷰 기준, 네이밍 컨벤션, 아키텍처 패턴이 개발자 재량에 맡겨져 있음.
==🔴**결과:** 코드베이스 일관성 저하, 유지보수 비용 증가, 온보딩 난이도 상승==

### 문제 3(테스트) — 단위·UI 테스트 자동화 부재
작성한 코드에 대한 단위 테스트·UI 테스트가 없음.
기능 변경 시 어디가 깨지는지 알 수 없어 전수 수동 확인에 의존.
==🔴**결과:** 기능 추가·수정마다 회귀 버그 위험 상존, 리팩토링 불가 구조==

### 결론 — 검증 비용이 전부 사람에게 집중됨
자동화 장치가 없으므로 구현 품질 검증이 **개발자 개인의 주의력**에만 의존.
기능이 늘어날수록 수동 검증 부담이 선형으로 증가하고, 테스트 커버리지는 반비례로 감소.
==🔴**결과:** 기능 추가 속도 저하 + 배포 후 핫픽스 반복 + 기술 부채 누적==

### 요약
| **문제** | **현재 방식** | **리스크** |
|:---:|:---:|:---:|
| 플랫폼 간 구현 불일치 | 개발자 개인이 기획서 해석 후 독립 구현 | 플랫폼별 동작 차이 → 재작업 |
| 코드 품질 기준 부재 | 개발자 재량 | 일관성 저하 → 유지보수 비용 증가 |
| 테스트 자동화 부재 | 전량 수동 QA | 회귀 버그 상존 → 핫픽스 반복 |

 → 개발자 개인 역량과 무관하게 일정 수준 이상의 코드 품질 보장

 → 기능 추가·수정 시 회귀 버그 자동 감지, 수동 QA 부담 감소

## 문제점 정의

### 근본 원인
**"기획서를 각 직군이 독립적으로 해석하고 작업을 시작한다"**
공통 해석 기준이 없는 상태에서 백엔드 / 디자인 / iOS / Android/ Web이 동시에 작업을 시작하기 때문에, 불일치가 만들어지는 게 구조적으로 피할 수 없음.

### 문제 1(기획) — 기획서 품질 문제
기획서 자체에 모호한 정책, 누락된 엣지케이스, 플랫폼별 예외 처리가 정의되지 않은 채로 내려옴.
하지만, 회의에서는 이걸 검토할 도구나 기준이 없음.
==🔴**결과:** 각자 빈칸을 스스로 채워서 해석==

### 문제 2(디자인) - 누락 및 불일치 디자인 문제
모호한 기획서를 바탕으로 디자인을 진행하여, 여러 엣지 케이스 대응 및 디자인 공통화가 지켜지지 않음.
==🔴결과: 문제 상황을 확인한 개발자로부터 수정요청==

### 문제 3(개발) — 공통 설계 기준 부재
회의에서 설계 논의가 없고, 회의 후 각 직군이 동일한 해석을 갖고 있는지 확인하는 방법이 없음.
용어, 데이터 모델, 비즈니스 로직을 누가 어떻게 해석했는지 아무도 모름.
==🔴**결과:** 백엔드 / iOS / Android/ Web이 같은 기능을 다르게 구현==

### 문제 4(백엔드) — API 스펙 검증
백엔드가 작업을 마친 후, Swagger로 API를 공유.
하지만, 표준 포맷이 없고 공유 시점도 늦음.
==🔴**결과:** 프론트엔드가 API 연동 전까지 실제 스펙을 알 수 없음==

### 문제 5(테스트) - 미흡한 테스트
모든 작업이 지연되어 테스트 기간이 짧아짐.
또한, 미흡한 테스트 시나리오로 엣지 케이스들에 대해 검증하지 못함.
==🔴**결과**: 배포 이후 버그 발견 또는 문의로 핫픽스 작업==

### 결론 — 불일치 발견 시점이 너무 늦음
위 네 가지 문제의 결과물이 전부 **프론트엔드 연동 단계에서 처음 충돌**
이 시점은 이미 백엔드 / 디자인 / iOS / Android 모두 작업이 완료된 후
==🔴**결과:** 3가지 재작업 루프 발생==
* API 해석 불일치 → 백엔드 재작업
* 엣지케이스 발견 → 기획/디자인 재작업
* iOS/Android/Web 구현 차이 → 양쪽 재작업
* 짧은 테스트 기간 -> 배포 이후 버그 발견 후 핫픽스
